From PartyRock to Real App

A Step-by-Step Guide for Non-Technical Users
🥗 → 🌐
Turn any AWS PartyRock AI app into a published website
using real AWS services — no coding required

Works with personal AWS accounts AND AWS lab/sandbox environments

Estimated time: ~2–3 hours complete | ~2 hours quick path (Steps 0–8)

Overview

You built a cool AI app on AWS PartyRock. Now you want to share it as a real website that anyone can visit — not just people with a PartyRock account.

Choose Your Path

Path A — Personal AWS Account
Create your own AWS account. Full control, permanent deployment, near-zero cost on free tier. Requires a credit card.
🧪 Path B — AWS Lab / Sandbox Environment
Using AWS Skill Builder, Vocareum, AWS Academy, or an event sandbox? No account creation needed — credentials are provided. App works during the lab session only.

Architecture

User's Browser
      │  HTTPS
      ▼
CloudFront (CDN + HTTPS)  →  S3 Bucket (index.html)
                                    │  API calls with x-api-key header
                                    ▼
                          API Gateway (REST, throttled)
                                    │
                          Lambda  →  Bedrock (Claude)
Supporting: DynamoDB (chat history) · CloudWatch (alarms + logs) · IAM (scoped)

Step 2 — Set Up Your AWS Account (20 min) — Path A

Using a lab sandbox? Skip this step and go to Step 2b.
You need a credit card to sign up. Everything stays within the AWS Free Tier — charges are near $0 for a personal app.

Create an AWS Account

  1. Go to aws.amazon.com → click Create an AWS Account
  2. Enter email, choose account name, verify email, create password
  3. Add payment info, verify phone, select Basic support — Free

Create IAM User (github-deployer)

  1. Sign in to console.aws.amazon.com → search IAM → Users → Create user
  2. Username: github-deployer → Attach policies directly → check these 6:
  • AWSCloudFormationFullAccess
  • AWSLambda_FullAccess
  • IAMFullAccess
  • AmazonAPIGatewayAdministrator
  • AmazonS3FullAccess
  • AmazonBedrockFullAccess

Create Access Key

  1. Click github-deployer → Security credentials tab → Access keys → Create access key
  2. Select Application running outside AWS → Create
  3. Save both values — Secret Access Key is only shown ONCE
The Secret Access Key is only shown ONCE. Save it now before closing the page.
ValueGitHub secret name
Access Key ID (starts with AKIA)AWS_ACCESS_KEY_ID
Secret Access KeyAWS_SECRET_ACCESS_KEY

Create S3 Bucket + Lifecycle Rule

  1. S3 → Create bucket → Name: yourname-sam-deploy-2024 → Region: us-east-1 → Create
  2. Click the bucket → Management tab → Create lifecycle rule → Name: expire-old-artifacts
  3. Apply to all objects → Expire current versions → 30 days → Create rule

  • IAM user github-deployer with all 6 policies
  • Access Key ID saved
  • Secret Access Key saved

Step 2b — Sandbox Environment Setup (20 min) — Path B

This step replaces Step 2 for users in AWS Skill Builder, Vocareum, AWS Academy, Workshop Studio, or event sandboxes.

Get Your Sandbox Credentials

In your lab panel, click AWS Details or Account Details. Copy all three values:

ValueFormatGitHub secret name
Access Key IDStarts with ASIAAWS_ACCESS_KEY_ID
Secret Access KeyLong random stringAWS_SECRET_ACCESS_KEY
Session TokenVery long string — copy ALL of itAWS_SESSION_TOKEN
Session tokens expire when your lab session ends. Your app will stop working after expiry. For a permanent app, use a personal AWS account.

Find LabRole ARN + Create S3 Bucket

  1. AWS Console → IAM → Roles → find LabRole → copy its ARN: arn:aws:iam::123456789012:role/LabRole
  2. S3 → Create bucket → Name: lab-sam-deploy-[initials]-[4 digits] → same region as lab

Use the Pre-Generated Sandbox Template

  1. Open Kiro, share your infra/template.yaml
  2. Ask: "Generate a template-sandbox.yaml that uses LabRole instead of creating an IAM role. Use YOUR_ACCOUNT_ID as a placeholder."
  3. Replace YOUR_ACCOUNT_ID with your actual 12-digit account ID (shown in top-right of AWS Console)
  4. Upload template-sandbox.yaml to your GitHub repo in the infra/ folder
  5. Update deploy.yml to use template-sandbox.yaml and remove CAPABILITY_NAMED_IAM

Add Session Token to deploy.yml

- name: Configure AWS credentials
  uses: aws-actions/configure-aws-credentials@v4
  with:
    aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
    aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
    aws-session-token: ${{ secrets.AWS_SESSION_TOKEN }}   # ← add this
    aws-region: ${{ vars.AWS_REGION }}                    # ← use variable
  • Credentials (Key ID + Secret + Session Token) copied
  • LabRole ARN copied
  • S3 bucket created
  • template-sandbox.yaml generated, ACCOUNT_ID replaced
  • deploy.yml updated with session token and region variable

Step 3 — Enable Bedrock + Tighten IAM + CloudWatch (25 min)

Enable Bedrock Model Access

  1. AWS Console → search Bedrock → confirm region is us-east-1 (top right)
  2. Left sidebar → Model access → Manage model access
  3. Under Anthropic, check Claude 3 Sonnet → Save changes
  4. Wait for status to show Access granted ✅
If Claude is not available in your lab, look for Amazon Titan Text Express (amazon.titan-text-express-v1) — usually pre-enabled. Update the model ID in your Lambda functions.

Tighten IAM Role to Specific Model Only

  1. IAM → Roles → search your app name → click the Lambda execution role
  2. Permissions tab → find the Bedrock policy → Edit
  3. Replace the policy statement with:
{ "Effect": "Allow", "Action": "bedrock:InvokeModel",
  "Resource": "arn:aws:bedrock:us-east-1::foundation-model/anthropic.claude-3-sonnet-20240229-v1:0" }
This scopes your Lambda to only call one specific model instead of any model in your account.

Create a CloudWatch Dashboard

  1. AWS Console → CloudWatch → Dashboards → Create dashboard → Name: my-app-overview
  2. Add widget → Line → Lambda → By Function Name → check Errors for each function → Create widget
  3. Add another widget → same steps but select Invocations
  4. Save dashboard
  • Claude 3 Sonnet shows "Access granted"
  • IAM role updated to specific model ARN only
  • CloudWatch dashboard created with Errors and Invocations widgets

Set Up API Key Protection (Step 7.5)

  1. AWS Console → API Gateway → click your API → Usage Plans → Create usage plan
  2. Name: my-app-plan · Rate: 10 req/s · Burst: 20 → Next
  3. Add API Stage → select your API → stage: prod → Next
  4. Create API Key and add to Usage Plan → Name: my-app-key → Auto Generate → Save
  5. Click the key → Show → copy the value → update GitHub API_KEY secret to match
  6. API Gateway → your API → each POST method → Method Request → API Key Required: true
  7. Actions → Deploy API → prod stage → Deploy
  1. Lambda → each function → Configuration → Concurrency → Reserve concurrency: 5 → Save

Set Up CloudWatch Alarms (Step 7.6)

  1. CloudWatch → Alarms → Create alarm → Lambda → By Function Name → Errors → threshold > 5
  2. Create new SNS topic → enter your email → confirm the subscription email AWS sends you
  3. Name: my-app-lambda-errors → Create alarm
  4. Repeat for API Gateway → By Stage → 5XXError → name: my-app-api-errors

Step 9 — Make It Production-Ready (20–45 min)

9.1 Add HTTPS with CloudFront

Recommended for everyone — gives you HTTPS for free and makes copy buttons work.
  1. AWS Console → CloudFront → Create a CloudFront distribution
  2. Origin domain: select your frontend S3 bucket (the .s3-website-us-east-1.amazonaws.com endpoint)
  3. Viewer protocol policy: Redirect HTTP to HTTPS
  4. Leave all other settings as default → Create distribution
  5. Wait 5–10 minutes for status to change to Enabled
  6. Copy the Distribution domain name (e.g. d1234abcd.cloudfront.net) — your new HTTPS URL

Then lock down S3 to CloudFront only: ask Kiro to generate an S3 bucket policy that only allows access from your CloudFront distribution ARN.

9.2 Add a Custom Domain (Optional)

  1. Certificate Manager (must be in us-east-1) → Request a certificate → enter domain → DNS validation → Create records in Route 53
  2. CloudFront → your distribution → Edit → add domain as Alternate domain name → select certificate → Save
  3. Route 53 → Hosted zones → your domain → Create record → type A → Alias → CloudFront distribution

9.3 Add AWS WAF Rate Limiting (Optional, ~$5/month)

  1. WAF & Shield → Create web ACL → Regional → us-east-1 → Name: my-app-waf
  2. Add managed rule: AWSManagedRulesCommonRuleSet
  3. Add own rule: Rate-based → 100 per 5 min → Block
  4. Attach to API Gateway: API Gateway → your API → Stages → prod → WAF tab → select ACL

9.4 DynamoDB Chat Persistence (Chatbot Apps Only)

  1. DynamoDB → Create table → Name: my-app-chat-sessions → Partition key: session_id (String)
  2. Ask Kiro to update your chatbot Lambda to store/retrieve conversation history with 24h TTL
  3. IAM → Lambda role → Add permissions → AmazonDynamoDBFullAccess
  4. Ask Kiro to update frontend to generate session_id and add a Clear Chat button

9.5 Large File Uploads via Presigned URL (File Upload Apps Only)

Default behaviour: The generated app reads files in the browser using FileReader, converts to base64, and sends directly in the Lambda request body. This works for files up to ~15MB (Lambda Function URLs support 20MB payloads; base64 adds ~33% overhead). Most photos, PDFs, DOCX, and CSV files fall well within this limit.

Only needed for files larger than ~15MB: Ask Kiro: "Add a presigned URL upload endpoint. Lambda generates a presigned S3 PUT URL, frontend uses it for files larger than 15MB. Store uploads in a new S3 bucket called my-app-uploads."

  • CloudFront distribution created, HTTPS URL working
  • S3 bucket policy updated to allow CloudFront only
  • (Optional) Custom domain connected
  • (Optional) WAF attached to API Gateway
  • (Optional) DynamoDB chat sessions table created

Step 10 — Update Your App After It's Live (5 min)

Every time you save a change to GitHub, your app redeploys automatically in 3–5 minutes.

Update a Prompt

  1. Open Kiro → share the Lambda file → describe the change → Kiro edits it
  2. GitHub → navigate to the file → pencil icon → select all → paste updated content → Commit changes

Update the Frontend

  1. Open Kiro → share frontend/index.html → describe the change → Kiro edits it
  2. GitHub → frontend/index.html → pencil icon → select all → paste → Commit changes

Roll Back a Broken Change

  1. GitHub → click the file → History (clock icon) → click the commit before your broken change
  2. View the old version → copy content → go back to current file → edit → paste → Commit
If the deployment itself failed, CloudFormation automatically rolls back. Check CloudFormation → your stack → Status column for ROLLBACK_COMPLETE.
After updating CloudFront-served files, force a browser refresh with Ctrl+Shift+R (Windows) or Cmd+Shift+R (Mac).

Step 11 — Delete Your App (Cleanup)

Deleting your app is permanent. Your GitHub repository and code are NOT affected.
  1. CloudFormation: Find your stack → check box → Delete → confirm. If it fails with "bucket not empty", empty the S3 bucket first (see step 3).
  2. SAM artifacts bucket: S3 → find yourname-sam-deploy-2024 → Empty → Delete
  3. CloudWatch log groups: CloudWatch → Log groups → select /aws/lambda/ groups → Actions → Delete
  4. CloudFront: CloudFront → select distribution → Disable first (wait 5–10 min) → then Delete
  5. DynamoDB table (if applicable): DynamoDB → Tables → select → Delete
  6. IAM user (optional): IAM → Users → github-deployer → Delete
CloudFront distributions MUST be disabled before they can be deleted. Trying to delete without disabling first will fail.

To redeploy later: GitHub → Actions → your workflow → Run workflow. Your app redeploys from scratch in 3–5 minutes.
